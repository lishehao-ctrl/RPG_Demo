# RPG_Demo 轻关系叙事转型定调

本文件回答的问题是：

- `如果不再坚持当前过重的人文 / 制度 / civic dossier 路线，RPG_Demo 最值得转向什么方向？`
- `这次转型里，哪些能力应该保留，哪些表达必须替换？`
- `新的产品核心体验应该是什么？`

这不是实现细节文档，而是一份产品方向与体验定调底稿。

## 结论

RPG_Demo 最值得转向的不是“纯陪聊恋爱”，也不是“继续深挖高门槛 civic procedural”。

最适合的方向是：

`轻乙女 / 都市情感悬疑 / 关系博弈叙事`

一句话版本：

`一款让用户进入、推进并改写人物关系走向的互动关系小说。`

更完整版本：

`一款围绕心动、误会、选择、站队、秘密与关系反转展开的互动叙事产品；它不是单纯陪聊，而是把剧情推进、角色关系与可编辑工作流做成了一个可玩的连续体验。`

## 为什么要转

当前版本的主要问题不是系统不成立，而是：

- `题材门槛过高`
  - 公共记录、授权、配给、见证、秩序这些关键词对普通用户的即时吸引力不够强
- `情绪驱动力偏弱`
  - 用户更容易欣赏“设定完整”，但不一定会产生“想继续玩”的冲动
- `UI 语气过冷`
  - 现在的档案馆 / 编辑部 / dossier 工作台更像高概念作品，而不是大多数用户愿意沉浸的互动娱乐产品

当前最有价值的并不是那层沉重题材，而是已经做好的底层系统：

- `story seed -> preview -> author -> publish -> play`
- Author Copilot 的可控编辑链路
- stateful runtime
- checkpoint / session 恢复
- LLM-as-a-Judge + Trace-based Evaluation

所以最优策略不是推倒重来，而是：

`保留系统能力，替换题材外壳与产品表达。`

## 新方向是什么

### 核心品类

新方向不应该叫：

- 泛故事引擎
- AI RPG 平台
- AI 陪聊恋爱

更合适的主品类表述是：

- `互动关系叙事产品`
- `可推进、可改写的互动关系小说`
- `带悬疑和路线感的都市情感叙事`

### 体验感受

用户打开后，最先感知到的应该不是：

- 制度
- 议会
- 档案
- 公共授权

而应该是：

- `我是谁`
- `我和谁会发生关系变化`
- `谁在试探我`
- `谁对我有秘密`
- `我下一步选谁 / 信谁 / 撩谁 / 逼谁 / 站谁`

### 情绪目标

这条产品线要追求的核心感受不是“严肃”和“正确”，而是：

- `上头`
- `追更感`
- `代入感`
- `人物张力`
- `关系反转`
- `名场面驱动`

## 它不应该变成什么

这次转型也有边界。

### 不做成纯甜宠

如果只剩撒糖、心动和甜度堆砌，产品会失去持续推进力。

我们要的是：

- 有情感推进
- 但也有误会、悬疑、立场冲突、秘密和代价

### 不做成纯陪聊

如果变成无限闲聊、情绪陪伴、没有剧情推进的产品，就会丢掉当前最强的系统能力。

RPG_Demo 的优势不在开放闲聊，而在：

- 有结构
- 有推进
- 有状态
- 有结局
- 有可验证性

### 不做成泛爽文

不能为了门槛低，就退化成：

- 随便堆设定
- 靠强反派和大反转刺激
- 只追求“下一句更炸”

新方向仍然要有：

- 连续剧情
- 角色一致性
- 情绪递进
- 关系反馈

## 新方向的核心体验

### 用户身份

新方向必须给用户一个更明确、更容易代入的主角身份。

主角不再是“公共系统里的承压节点”优先，而应优先是：

- 新入职的关键人物
- 关系网络中的中心角色
- 被多个强角色拉扯的人
- 拿着某个秘密、身份或误会进入局中的人

理想状态是：

`用户一进入故事，先想的是“我会卷进谁和谁之间”，而不是“这个制度结构如何运作”。`

### 角色配置

每个故事最好固定有 `3-5` 个强角色对象。

这些角色必须具备：

- 强人设
- 可推进关系
- 各自的秘密
- 不同立场
- 不同吸引点

他们之间的关系不只是“对主角有好感”，还应该包含：

- 误会
- 试探
- 旧账
- 利益冲突
- 站队变化

### 每回合推进什么

当前版本每回合更像在推进公共危机。

新方向里，每回合更应该推进的是：

- `关系`
- `误会`
- `选择`
- `站队`
- `亲密度`
- `信任`
- `秘密曝光`

也就是说，用户不是在“处理制度问题”，而是在：

- 靠近谁
- 怀疑谁
- 利用谁
- 安抚谁
- 逼问谁
- 公开谁的秘密

## 题材范围

### 最推荐的主方向

- `都市情感悬疑`
- `轻乙女`
- `关系博弈叙事`

### 可承接的世界观外壳

这些都可以做，但它们只是壳，不是核心：

- 职场
- 校园
- 娱乐圈
- 豪门
- 都市异能
- 轻悬疑

要求是：

`世界观必须服务于人物关系，而不是反过来让人物只服务设定。`

### 最适合的剧情关键词

- 心动
- 误会
- 试探
- 站队
- 嫉妒
- 信任
- 表白前夜
- 名场面
- 公开秘密
- 关系反转

## 当前系统里什么可以保留

### 必须保留

- `完整工作流`
  - seed -> preview -> author -> publish -> play
- `Author Copilot`
  - 继续做成剧情导演 / 编剧室
- `stateful runtime`
  - 继续保留 session、history、结局判断、恢复能力
- `evaluation`
  - 继续保留 benchmark、judge 和 trace

### 这些是资产，不是包袱

当前真正值钱的是：

- 系统闭环
- 可控编辑
- 可验证质量

这些能力迁到更大众的题材上，只会更强，不会更弱。

## 当前系统里什么必须替换

### 1. 品牌语言

要从：

- 档案
- 授权
- 配给
- 见证
- 秩序

切到：

- 心动
- 秘密
- 信任
- 站队
- 误会
- 名场面

### 2. runtime 指标语言

现在更像：

- civic pressure
- legitimacy
- public consequence

新方向建议替换成：

- `亲密度`
- `信任度`
- `危险感`
- `秘密暴露度`
- `公开形象`
- `站队倾向`

### 3. 世界推进逻辑

现在更像“公共系统如何在压力下做决定”。

新方向应该变成：

`人物关系如何在压力下不断重排。`

### 4. UI 气质

现在的 UI 更像：

- 档案馆
- 社论页
- 制度工作台

新方向应该更像：

- 连载剧集
- 关系板
- 私密消息与事件回放
- 名场面卡片
- 情绪张力更强的都市戏剧界面

## UI 应该怎么改

### 不用推倒重来

当前很多结构是可以保留的：

- 两栏/多面板布局
- Story library
- Story detail
- Play session
- Copilot 面板
- diff / preview / apply 这些交互骨架

### 但表达层必须重做

要改的不是系统骨架，而是：

- 视觉气质
- 文案体系
- 信息架构优先级
- 状态反馈语义

### 新 UI 的推荐方向

不是二次元甜宠粉色风，也不是冷档案室。

更建议走：

`高级都市戏剧 + 关系悬疑`

参考感受：

- 连载剧集感
- 手机屏 / 聊天记录 / 事件回放
- 角色卡 / 关系图
- 夜色城市 / 霓虹 / 酒会 / 会议室 / 后台 / 校园夜路 / 私密房间
- 有一点危险感和暧昧感，但不低幼

### 页面语义映射

- `Library`
  - 从“故事库”变成“剧集 / 路线入口”
- `Story Detail`
  - 从“档案详情”变成“作品页 + 角色关系 + 入坑理由”
- `Play Session`
  - 从“局势推进面板”变成“剧情推进 + 关系波动 + 关键选择”
- `Author Copilot`
  - 从“编辑助手”变成“剧情导演 / 编剧室”

## Copilot 应该怎么转

这是这次转型里最有潜力的模块之一。

从现在的：

- 改人物设定
- 改剧情骨架
- 改结局倾向
- 改玩法规则

升级成用户更好理解的表述：

- 改感情线节奏
- 改关系站队
- 改角色出场张力
- 改名场面设计
- 改修罗场结构
- 改结局倾向
- 改台词风格

也就是说，Copilot 不再只是技术上的编辑器，而是：

`剧情导演 / 编剧室`

## 竞争力怎么重新表述

转型之后，竞争力不应该再主要围绕“公共危机题材”来讲。

应该改成：

### 1. 不只是甜，不只是聊，而是可推进的关系叙事

很多同类产品只能陪聊，或者只能生成一段暧昧文本。

RPG_Demo 更强的地方是：

- 有剧情推进
- 有关系反馈
- 有路线张力
- 有结局与反转

### 2. 用户可以玩，也可以改

大部分产品只能消费内容。

RPG_Demo 可以：

- 玩剧情
- 改剧情
- 调感情线
- 调角色关系
- 预览差异

### 3. 有状态化 runtime，不是无限闲聊

不是一句句陪你聊，而是每一步都有关系后果和局势变化。

### 4. 有评测闭环，能知道“哪里上头、哪里无聊”

这会直接帮助新方向成立，因为关系叙事最大的风险就是：

- 甜得太空
- 反转太假
- 台词重复
- 人设漂移

有 LLM-as-a-Judge + trace，这些可以被定位，不是只能靠感觉。

## 这条转型路线为什么适合当前项目

原因很简单：

- 现有 `authoring + runtime + evaluation` 能力都能直接复用
- 但用户体感会立刻变轻
- 更容易做出“上头”“追更感”“代入感”
- 更容易让普通用户一眼知道怎么玩

也就是说：

`你不是系统不够强，而是当前题材壳子把你的系统能力包得太重了。`

## 推荐的最终定调

如果现在就要选一句最能代表新方向的话，我推荐：

`RPG_Demo 将从制度危机叙事，转向以人物关系、秘密与选择为核心的互动关系小说。`

如果更产品化一点，可以写成：

`一款让用户一边推进剧情、一边改写人物关系走向的互动叙事产品。`

如果更面向普通用户一点，可以写成：

`不是陪聊，而是让你亲手推着角色关系失控、靠近、反转的互动小说。`

## 下一步建议

如果继续往下做，优先级建议是：

1. 先定新题材与角色模板
2. 再重写 runtime 状态语义
3. 再改 UI 文案和视觉表达
4. 最后再做 benchmark 维度切换

原因是：

- 先改系统壳子，比先改美术更关键
- 先把“玩什么”定义清楚，UI 才不会乱
- benchmark 需要跟着新体验目标一起改，不然还是在拿旧指标测新产品
