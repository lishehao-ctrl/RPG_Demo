# Tiny Stories Portfolio Demo Video Script

## Goal

Create a portfolio-ready demo video that proves Tiny Stories is a real playable system, not only a concept trailer.

The final video should mix three layers:

1. Cinematic AI-generated frames for atmosphere, emotional transitions, and chapter breaks.
2. Real product capture showing seed input, generation feedback, player choices, free-form action, advisor chat, state changes, and ending share.
3. Motion overlays that explain the system briefly without interrupting the gameplay.

Target length: 75-90 seconds.

Tone: dark Korean webtoon drama, clear product proof, fast but readable.

## Core Narrative

The viewer should understand this in one pass:

> I type one dramatic seed. The system turns it into a playable drama with characters, hidden goals, live state, choices, free input, an advisor, and a shareable ending.

## Demo Story Seed

Use one polished English seed throughout the video:

> At my wedding, the groom asks me to sign away my shares before the ceremony starts.

Backup seed if the live run needs stronger corporate stakes:

> Minutes before the awards livestream, my cofounder announces a secret merger that cuts me out.

Preferred first seed because it matches the current wedding/manhwa keyframes and has immediate emotional clarity.

## Editing Rules

Do:

- Let real UI prove the product.
- Use AI art as chapter punctuation and emotional glue.
- Keep every UI text crop readable for at least 1.2 seconds.
- Show at least one typed seed and one typed free-form action.
- Show at least one system feedback moment after the player acts.
- Preserve English UI throughout.

Do not:

- Make it only a cinematic mood trailer.
- Hide all product UI behind heavy blur or overlays.
- Over-explain with paragraphs of text.
- Show generated artwork pretending to be gameplay.
- Spend more than 5 seconds on any one static screen unless typing is happening.

## Capture Notes

Recommended capture size:
1920x1080 desktop browser.

Recommended product route:

1. Start at `#/create`.
2. Create the wedding seed.
3. Use `Gauntlet mode` if the backend run is stable enough.
4. Continue until at least one consequence is visible.
5. If the live ending takes too long, use an existing completed session for the ending proof, but keep the seed/run visually consistent.

Recommended motion in Remotion:

- Use real screen recordings as `<Video>` layers.
- Use generated keyframes as full-screen `<Img>` transition layers.
- Use masked crop windows to zoom into input, role, choices, advisor, and ending.
- Add a progress ribbon at the top of the video showing:
  `Seed -> Build -> Play -> Advise -> Consequence -> Ending`

## Version 2 Improvement Over Current Trailer

The current trailer mostly communicates the concept. This version should communicate:

1. I can type into the product.
2. The product responds with generated playable structure.
3. I can make fixed and free-form moves.
4. The UI updates with consequence and state.
5. The ending is a result of play, not a static mockup.

That makes the video stronger for a personal portfolio because it demonstrates both product thinking and engineering implementation.
